Architecture Approach

How should the app connect to your services?

A

Direct API — Flutter talks to everything

The Flutter app connects directly to Immich, Music Assistant, Home Assistant, and Frigate APIs via REST + WebSocket. No middleware.

Pros

  • Simplest to build and deploy — one app, no backend
  • Lowest latency — direct WebSocket connections
  • Fewer moving parts to break

Cons

  • All API logic lives in Dart — switching services means app changes
  • Credentials stored on-device
  • Harder to share state if you build a second hub later
B

Thin Backend — Local API gateway on the Pi

A lightweight Go/Node service on the Pi aggregates all APIs into one unified WebSocket. Flutter connects only to localhost.

Pros

  • Clean separation — Flutter is pure UI, backend handles API logic
  • Easy to swap services without touching the app
  • Backend can cache photos, pre-fetch memories, handle Frigate events
  • Voice integration later plugs into the backend naturally

Cons

  • Two things to build and maintain
  • More RAM usage on Pi (~50-100MB for the backend)
  • Added complexity for initial development
C

HA-Centric — Home Assistant as the hub

Route everything through Home Assistant. Use HA's Music Assistant, Frigate, and Immich integrations. Flutter only talks to HA's WebSocket API.

Pros

  • Single API connection to manage
  • Leverages existing HA integrations you already have
  • HA handles automations (doorbell → show camera) server-side

Cons

  • HA becomes a single point of failure for everything
  • Limited control over Immich photo selection (HA integration is basic)
  • Higher latency for camera feeds and music control
  • You're back to depending on HA's UI paradigm for data

Recommendation: A (Direct API)

For a single-hub household, Direct API is the right starting point. It's the fastest path to a working product, and the "harder to scale" downside doesn't matter until you build hub #2. The architecture naturally allows extracting a backend later if needed — the API client classes become the backend's logic layer. Voice integration (when you add it) can run as a sidecar process that sends commands to the same APIs.
